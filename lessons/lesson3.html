Platform Architecture

Layers of a Computing Platform

  • Hardware: Physical components such as CPU, memory, storage, and peripherals
  • Firmware: BIOS/UEFI, device firmware, and initialization processes
  • Operating System (OS): Manages hardware, runs applications, enforces security
  • Applications: Software that provides user-level services

Mobile vs. Desktop Platforms

Mobile platforms are optimized for portability and battery life. They integrate hardware and software tightly and use app sandboxing with stricter permissions (Android/iOS).

Desktop platforms focus on higher performance and flexibility. They have a broader software ecosystem and more customization options, but also a larger attack surface.

Cloud and Virtualization as Platforms

Cloud platforms abstract physical resources and provide services on demand (AWS, Azure). Security is based on the shared responsibility model.

Virtualization platforms use hypervisors to enable multiple virtual machines on one physical machine. This increases resource efficiency but also introduces risks like VM escape and misconfigurations.

Sample Platform Attack Surface

  • Hardware: Side-channel attacks, physical tampering
  • Firmware: BIOS/UEFI rootkits, firmware backdoors
  • Operating System: Privilege escalation, kernel exploits
  • Applications: Malware, unpatched vulnerabilities

Trusted Platform Module (TPM)

TPM is a hardware-based security chip that provides a root of trust for a computing platform. It works independently of the main CPU and OS, ensuring secure storage, encryption, and attestation.

Why TPM Matters

  • Stores important digital keys, certificates, and passwords securely
  • Helps computers start safely through Secure Boot
  • Blocks hidden attacks during startup, including firmware or boot-level attacks

UEFI (Unified Extensible Firmware Interface)

UEFI is the modern replacement for BIOS. It initializes hardware when a computer powers on, then hands control to the operating system. It acts as a bridge between hardware and the OS.

Chain of Trust

Each stage of the boot process checks the integrity of the next stage:

  • Root of Trust (TPM/firmware key)
  • Firmware checks bootloader
  • Bootloader checks OS kernel
  • OS verifies drivers and applications

If any stage fails, the boot process stops or raises alerts.

TPM Versions

  • TPM 1.2: Basic cryptographic support (SHA-1, RSA)
  • TPM 2.0: Stronger algorithms (SHA-256, ECC), required for Windows 11 and modern enterprise security solutions

TPM Security Benefits

  • Protects against rootkits, bootkits, key theft, and unauthorized firmware changes
  • Provides hardware-enforced trust instead of relying only on software

TPM Limitations

  • Requires hardware support
  • TPM failure may result in loss of access to encrypted data
  • Cannot prevent OS-level malware once the system is running
  • Still vulnerable to advanced physical attacks on the chip

Virtualization

Virtualization allows multiple virtual machines (VMs) to run on one physical server.

Key features:

  • Consolidated hardware: One server hosts many virtual environments
  • Isolated environments: Each VM runs independently with its own OS and apps
  • Maximized utilization: Efficient use of hardware resources

Types of Virtualization

  • Server virtualization: Multiple virtual servers on one physical machine
  • Desktop virtualization: Virtual desktop infrastructure (VDI), remote desktop access
  • Network virtualization: Virtual networks independent of physical hardware
  • Storage virtualization: Pooled storage from multiple devices
  • Application virtualization: Applications run in isolated environments

Types of Hypervisor

  • Type 1 (Bare-Metal): Runs directly on hardware, high performance and security (VMware ESXi, Microsoft Hyper-V)
  • Type 2 (Hosted): Runs on top of a host operating system (VirtualBox, VMware Workstation)

Benefits of Virtualization

  • Cost reduction
  • Increased agility
  • Enhanced disaster recovery
  • Improved management
  • Easier testing and compatibility

Popular Virtualization Platforms

  • Enterprise: VMware vSphere, Microsoft Hyper-V, Citrix XenServer, Red Hat Virtualization
  • Desktop/Development: VMware Workstation/Fusion, Oracle VirtualBox, Parallels Desktop, QEMU/KVM

Security Risks of Virtualization

  • Hyperjacking: Attackers compromise the hypervisor itself
  • VM Escape: Malicious code inside a VM breaks isolation and accesses the host
  • Misconfiguration vulnerabilities: Weak credentials, overprivileged users, poor network segmentation, overcommitted resources

VM Isolation Techniques

  • Strong separation between VMs
  • Controlled resource allocation (CPU, RAM, storage quotas)
  • Access control to prevent unauthorized VM-to-VM communication
  • Snapshot monitoring to detect rollback attacks

Secure VM Networking

  • Virtual switches to control traffic
  • Network segmentation using VLANs
  • Firewall rules per VM or segment
  • IDS/IPS integration to detect malicious traffic

VM Snapshots and Rollback Security

Snapshots save system state for recovery or testing.

Risks include:

  • Rollback to vulnerable versions
  • Exposing sensitive data

Best practices: Encrypt snapshots, monitor creation, and patch after rollback.

Best Practices for Virtualization Security

  • Keep hypervisors updated and patched
  • Enforce strict access controls
  • Follow CIS and NIST security baselines
  • Monitor VM behavior with logs and alerts
  • Encrypt VM images and storage
